# airlock-deploy

The deployment companion to `airlock`. This file is the glossary specific to *this* project's terminology. For the underlying Airlock contract terms (Publisher, Consumer, A2A, A2O, Promise, Estimate, Contract), see [`Okohedeki/airlock` CONTEXT.md](https://github.com/Okohedeki/airlock/blob/main/CONTEXT.md).

## Language

**Target**:
The cloud platform we ship a publisher's agent to. v1: Cloudflare Workers only. Future targets are added one prescriptive recipe at a time; we don't ship abstractions until we have at least two real targets.
_Avoid_: Provider, host, vendor

**Recipe**:
The opinionated, prescriptive configuration we ship per target. A recipe owns secrets handling, custom-domain wiring, `/.well-known/` serving, logging hookup, and the deploy command. Recipes are tightly coupled to one target by design.
_Avoid_: Template, preset, blueprint

**Scaffold**:
The starter project we generate for a publisher. Includes a stub `contract.yaml`, codegen'd handler stubs, the chosen recipe's config files, and instructions. The scaffold is the publisher's, not ours — they edit it freely.
_Avoid_: Boilerplate, skeleton (we use scaffold consistently)

**Bundle**:
The static output of `airlock build` (machine spec + HTML docs + `llms.txt` + landing page). The bundle is an input to `airlock-deploy`, never modified by it.
_Avoid_: Build, dist, artifacts

**Handler stub**:
A typed function generated by `airlock codegen` from a skill's input/output schema. The publisher fills in the body; the recipe wires it to the runtime.
_Avoid_: Endpoint, route handler, function

**Wrangler / target CLI**:
The underlying platform's command-line tool. `airlock-deploy` wraps the target CLI rather than reimplementing it. Secrets and traffic never touch `airlock-deploy`.
_Avoid_: Adapter (that term belongs to airlock's channel adapters), driver

## Flagged ambiguities

**"Deploy"** has two valid uses:
- The verb: `airlock-deploy deploy` invokes the target CLI to push code.
- The project: `airlock-deploy` is the name of this whole repo.
Both are fine; context disambiguates. When clarity matters, use `airlock-deploy deploy` for the verb.

**"Runtime"** is forbidden as a description of this project. `airlock-deploy` is not a runtime; it is a deploy orchestrator. Saying "the airlock-deploy runtime" would imply we host the agent — we never do. See ADR-pending-0001 (to be written when the project actually starts).
